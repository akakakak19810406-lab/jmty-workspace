# team-info 概要説明マニュアル（まずはこちらをお読みください）

このファイルは、`team-info` 全体の入口マニュアルです。  
今のフォルダ構成、運用ルール、よく使うスキルの役割をまとめています。

## 1. まず覚えること

新しいパソコンでは、**まず setup ファイルを 1 回実行**してください。  
ここを飛ばすと、Python や Node.js や ffmpeg などの土台が足りず、あとで止まりやすくなります。

### 1-1. 最初に実行するコマンド

#### macOS

```bash
bash ./setup/setup_all.cmd
```

#### Windows

```powershell
.\setup\setup_all.cmd
```

- この最初のコマンドだけは、`team-info` のリポジトリルートを開いた状態で打つ前提です。
- `TEAM_INFO_ROOT` は、まずそのカレントディレクトリを見て決まります。

### 1-2. この setup ファイルで入るもの

- Python 3.11.9 と Docker ランタイム `team-info/python-skill-runtime:3.11.9`
- Node.js
- Codex CLI
- ffmpeg
- Tesseract OCR
- Remotion 用の Python パッケージ
- VOICEVOX MCP / Canva 補助 / Dify 用の npm パッケージ
- `TEAM_INFO_ROOT` の保存
  - macOS は `~/.config/team-info/env.sh` と `launchctl`
  - Windows はユーザー環境変数
- セットアップ最後の自動検証
  - `TEAM_INFO_ROOT`
  - `codex`
  - host fallback Python
  - Remotion / VOICEVOX MCP などの npm 依存
  - Docker ランタイム内の Python import / ffmpeg / tesseract

### 1-3. setup が終わったあとにやること

1. ターミナルをいったん閉じて、もう一度開きます。
   - macOS は GUI アプリにも反映するため `launchctl` にも保存されます。
2. `TEAM_INFO_ROOT` が入っているかを見ます。

#### macOS

```bash
echo "$TEAM_INFO_ROOT"
```

#### Windows

```powershell
echo $env:TEAM_INFO_ROOT
```

3. パスが出て、さらに setup の最後が `Verify status: passed` なら土台はそろっています。
4. 何も出なければ、作業場所の登録だけ手でやります。
   - macOS では `~/.config/team-info/env.sh` と `~/.zprofile` / `~/.zshrc` を確認します。

#### macOS

```bash
python3 "[team-info を置いた絶対パス]/.agent/skills/common/scripts/team_info_runtime.py" setup-local-machine --repo-root "[team-info を置いた絶対パス]"
```

#### Windows

```powershell
py -3 "[team-info を置いた絶対パス]\.agent\skills\common\scripts\team_info_runtime.py" setup-local-machine --repo-root "[team-info を置いた絶対パス]" --shell powershell
```

5. このパソコンをオーナー機として使うときだけ、最後に `--owner` を付けます。
6. Git の push 前のオーナー確認は、人に聞かず、`team_info_runtime.py owner-status` でこのパソコンを見て判断します。
7. setup が終わると、「このパソコンは一度 team-info で作業した」と分かる印が自動で保存されます。
   - macOS / Linux: `~/.config/team-info/worked_before_machines.json`
   - Windows: `%APPDATA%\team-info\worked_before_machines.json`
8. setup が終わったら、このファイルを最初からもう一度読み直してください。

## 2. setup ファイルだけでは終わらないもの

`setup_all.cmd` は、コマンドで入れられるものをできるだけ入れます。  
でも、**外部サービスの鍵** までは自動で終わりません。  
特に大事なのは、`Docker Desktop`、`VOICEVOX Engine`、`Canva の鍵ファイル`、`Codex CLI の初回認証` です。

### 2-1. Docker Desktop を入れる

Docker は、`Dify` や `n8n` を動かすための土台です。

#### 手順

1. ブラウザで `https://www.docker.com/products/docker-desktop/` を開きます。
2. 自分のパソコンに合う `Docker Desktop` をダウンロードします。
3. ダウンロードしたファイルを開いて、画面の案内どおりに入れます。
4. インストールが終わったら `Docker Desktop` を起動します。
5. アプリの画面で、Docker が動いている表示になるまで待ちます。
6. ターミナルを開いて、次のどちらも通るか見ます。

```bash
docker --version
docker compose version
```

7. バージョンが出れば、Docker の準備はできています。

#### Dify を起動する手順

1. Docker Desktop を起動したままにします。
2. 次のコマンドを実行します。

macOS:

```bash
bash "[team-info を置いた絶対パス]/run.sh" --project dify -d
```

Windows:

```powershell
& "[team-info を置いた絶対パス]\run.ps1" -Project dify -d
```

3. 少し待ってから、ブラウザで `http://localhost:3000` を開きます。
4. 画面が出れば Dify は使えます。

#### よくある止まり方

- `Cannot connect to the Docker daemon` と出る  
  -> `run.sh` / `run.ps1` を使えば、自動で Docker Desktop の起動確認と待機をします。
- `localhost:3000` が開かない  
  -> まだ起動中のことがあります。少し待ってから、もう一度開きます。

### 2-2. VOICEVOX Engine を起動する

VOICEVOX は、台本を音声にするときに使います。  
このリポジトリでは、**GUI アプリではなく Docker 上の VOICEVOX Engine** を使います。

#### 手順

1. Docker Desktop を起動したままにします。
2. 次のコマンドを実行します。

```bash
python3 "[team-info を置いた絶対パス]/.agent/skills/common/scripts/team_info_runtime.py" start-voicevox-engine
```

3. ブラウザで `http://127.0.0.1:50021/docs` を開きます。
4. ページが開けば、VOICEVOX Engine は動いています。

#### 止まったとき

- `http://127.0.0.1:50021/docs` が開かない  
  -> `start-voicevox-engine` が失敗していることが多いです。Docker Desktop を起動したまま、もう一度実行します。

### 2-3. Canva の鍵ファイルを書く

Canva を使うときは、`CANVA_CLIENT_ID` と `CANVA_CLIENT_SECRET` が必要です。  
setup ファイルは**空のひな形ファイル**までは作りますが、中身は自分で書く必要があります。

#### 手順

1. 次のファイルを開きます。
   - macOS: `"$HOME/.secrets/canva_credentials.txt"`
   - Windows: `"$HOME\\.secrets\\canva_credentials.txt"`
2. 中に次の 2 行があるか見ます。

```text
CANVA_CLIENT_ID=
CANVA_CLIENT_SECRET=
```

3. `=` の右側に、自分の Canva の値を書きます。
4. 上書きして保存します。
5. Canva 認証をしたいときは、次を実行します。

#### macOS

```bash
python3 "[team-info を置いた絶対パス]/.agent/skills/common/scripts/team_info_runtime.py" run-remotion-python -- "[team-info を置いた絶対パス]/mcp-servers/canva_auth.py"
```

#### Windows

```powershell
py -3 "[team-info を置いた絶対パス]\.agent\skills\common\scripts\team_info_runtime.py" run-remotion-python -- "[team-info を置いた絶対パス]\mcp-servers\canva_auth.py"
```

6. ブラウザが開いたら、Canva にログインして許可します。
7. 終わると、`canva_tokens.json` がホームフォルダの `.secrets` に保存されます。

#### 止まったとき

- `Canva の ID または秘密の文字が見つかりません` と出る  
  -> `canva_credentials.txt` の 2 行が空のままです。ID と秘密の文字を書いて保存し、もう一度やります。

### 2-4. rclone を使って Google ドライブに送る

Google Drive へファイルを送るときは、`rclone` を使います。
`setup` を終えると `rclone` は入っています。初回だけ `rclone config` で `gdrive` という名前の接続先を作ります。

1. `rclone version` を実行して、入っていることを確認します。
2. `/gdrive` を初めて使うときに出る案内にそって、`rclone config` で `gdrive` を作ります。
3. 使い方の細かい流れは `/gdrive` の説明を見ます。

> Google Drive for Desktop は使いません。

### 2-6. 困ったときに見る場所

- setup 全体の説明: `setup/README.md`
- Docker の説明: `docker/README.md`
- VOICEVOX の説明: `Remotion/Voicebox_TTS_Skill_Guide.md`

### 2-7. それでもわからないとき

1. まず、このファイルを最初からもう一度読みます。
2. どこで止まったか分かるように、画面のスクリーンショットを撮ります。
3. 次の Discord を開きます。  
   `https://discord.com/channels/1478351976168165511/1479287635535990794`
4. 「どの手順のどこで止まったか」と「スクリーンショット」を一緒に送ります。
5. これで、見た人が状況をつかみやすくなります。

## 4. 主要フォルダの役割

- `.agent/skills/`
  - AI スキル本体。`SKILL.md` と必要な `scripts/` や `assets/` が入ります。
- `inputs/`
  - 入力素材の置き場です。生成物は置きません。
- `outputs/`
  - 共有で扱う生成物の正本の出力先です。個人専用の成果物は `personal/<account>/outputs/` に分けます。
  - AcoRiel 動画: `outputs/acoriel/renders/`
  - AcoRiel 概要欄: `outputs/acoriel/descriptions/`
  - Sleep Travel 音声: `outputs/sleep_travel/audio/`
  - Sleep Travel 動画: `outputs/sleep_travel/renders/`
- `personal/<account>/outputs/`
  - 個人専用の生成物・レポート・LP下書き・プロフィール素材を置く場所です。
  - ジモティー: `outputs/jmty/factory/` と `outputs/jmty/remote/`
  - バズ動画解析: `outputs/viral-analysis/`
- `Remotion/`
  - 動画制作と音声生成の作業場所です。
  - 実際のレンダリング実装は `Remotion/my-video/` にあります。
- `personal/<account>/`
  - 個人開発モードの保存先です。
  - 個人用 Apps Script は `personal/<account>/gas/` に置きます。
  - 個人用の補助スクリプトや定期実行物は `personal/<account>/scripts/` に置きます。
- `GAS/`
  - チームで共有する Google Apps Script の置き場です。
- `マニュアル/`
  - 人が読むための入口です。正本のルールは `AGENTS.md` と `RULES.md` を参照します。

## 5. 出力先ルール

- 出力先は常にルートの `outputs/` を使います。
- `Remotion/output/` `Remotion/renders/` `Remotion/my-video/out/` `Remotion/my-video/outputs/` は使いません。
- AcoRiel の概要欄は `outputs/acoriel/descriptions/` に保存します。
- レンダリングコマンドは、できるだけ次を使います。

```bash
bash "$TEAM_INFO_ROOT/Remotion/scripts/render_to_outputs.sh" <CompositionId> <output-file>.mp4
```

- `npx remotion render` を直接使うときも、出力先は `"$TEAM_INFO_ROOT/outputs/<channel>/renders/"` に固定します。

## 7. 運用ルール

- 新規スキル追加時は、`AGENTS.md` のスキル一覧も更新します。
- 人向けマニュアルを直すときは、先に `AGENTS.md`、`RULES.md`、各 `SKILL.md` の内容を確認して合わせます。
- 入力は `inputs/`、生成結果は `outputs/` に分けて管理します。
- 大きい変更や削除、push の前は、必ずユーザー確認を取ります。
- ユーザーに選択や確認を求める前は、通知を鳴らします。
- Claude Code 標準機能（`/loop`、`/schedule`、`/btw`、`/branch`、`/remote-control`、`/mobile`、`/hooks` など）と、team-info 独自 slash command は別物として扱います。
- Claude Code の `/voice` は標準の音声入力です。VOICEVOX で台本を音声化したいときは、`voice-script-launcher` を自然文で明示します。
- repo 共有の hook は `.claude/settings.json` に入れ、短時間・read-only・外部送信なし・自動編集なし・Docker / サーバー自動起動なしを守ります。
- `SessionStart` hook は、`worked-before-status` と `.dev-mode` のような短い確認だけに使います。
- `/loop` と `/schedule` は、読み取り・見守り・リマインド用途を基本にし、ファイル編集・Git・Docker・外部 API を定期自動実行する前は確認を取ります。
