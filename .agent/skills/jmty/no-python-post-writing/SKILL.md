# 投稿文作成で Python を使わないルール

## 絶対ルール
- 投稿文の本文作成、再作成、リライトを Python の固定テンプレートや文字列連結で行わない。
- Python は、入力読取、対象抽出、プロンプト組立、AI実行、保存、検証、スプレッドシート反映の補助だけに使う。
- 投稿文そのものは、Codex/AI に案件素材、現在の投稿文、投稿文ルール、スタイル見本を渡して生成する。
- 工場・在宅の共通処理を作る場合でも、在宅投稿文を Python テンプレートに戻さない。

## 在宅投稿文で参照する素材
- `inputs/jmty_remote_samples/`
- `inputs/jmty_post_style_samples/remote/`
- スプレッドシート `アカウント情報` の在宅投稿文セル
  - 在宅1: `S列`
  - 在宅2: `U列`
- GUIまたはローカルに保存済みの投稿文
- GUIで保存されている投稿文作成ルール

## 在宅投稿文の必須条件
- `完全在宅` を必ず入れる。
- `未経験OK` を必ず入れる。
- CTA は `【公式LINEURL】` を使う。
- 都道府県名、市区町村名、駅名などの地名を入れない。
- 投稿文にある職種、または在宅職種リストから選んだ職種に合わせて内容を変える。
- 全アカウントで、タイトル、冒頭、訴求、対象人物像、構成、CTA前の流れを重複させない。

## 実装時の判断
- 既存コードに `build_post_text()` のような Python本文生成関数がある場合、投稿文作成フローでは使わない方向へ置き換える。
- Python関数を残す場合は、失敗時の緊急フォールバックやテスト用に限定し、通常の週次/GUI投稿文作成では呼ばない。
- AI生成後は、Pythonで次の検証だけ行ってよい。
  - `完全在宅` があるか
  - `未経験OK` があるか
  - `【公式LINEURL】` があるか
  - 実URLが混入していないか
  - 在宅投稿文に地名が混入していないか
  - 途中切れや重複タイトルがないか

## 週次処理への適用
- 週次処理で投稿文を作る場合も、GUIのAI再作成と同じ考え方にする。
- スプレッドシートの在宅元文をそのまま整形するだけでなく、在宅案件素材とスタイル見本をAIへ渡して生成する。
- 工場投稿文も、ユーザーが求める場合は同じ方針でAI生成へ寄せる。ただし既存依頼が在宅限定なら工場処理は触らない。

## 完了条件
- 投稿文作成フローの説明で「Pythonで本文を作る」と言わない。
- コード変更時は、通常フローでPython固定本文生成が呼ばれないことを確認する。
- 変更後は構文チェックと、少なくともプロンプト組立または対象抽出の小テストを行う。
